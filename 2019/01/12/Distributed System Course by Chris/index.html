Distributed System Course by Chris

这个课程是Youtube上一个比较简短且高质量的分布式系统学习课程，由Chris Colohan提供。在开始学习MIT 6.824之前，通过该课程简单建立对分布式系统的概念，建立基本的知识框架。

课程主页：http://www.distributedsystemscourse.com

以下为课程笔记。前两节课是介绍分布式系统的What and Why，对分布式系统的正式介绍内容从第3节开始。

L3: How to learn distributed systems

Topics in distributed systems

  • How system fail
  • How to express your goals: SLIs, SLOs and SLAs
  • How to combine unreliable components to make a more reliable system
  • How nodes communicate —— RPCs
  • How nodes find each other —— naming
  • How to use time in a distributed world
  • How to get agreement —— consensus
  • How to persist data —— distributed storage
  • How to secure your system
  • How to operate your distributed system —— the art of SRE

L4: What could go wrong?

本节课程主要介绍了分布式系统中可能会出现的各种bug，如：

  • Crash or CrashLoop：循环崩溃
  • Data Corruption：数据错误
  • Data Loss：数据丢失
  • Server Down：服务器宕机
  • Query of Death：死亡队列（和DoS啥区别？）
  • Broken Dependency：信赖的服务器向你发来了错误的答案
  • DoS Attack：请求太多，超出服务器负载
  • Cascading Failure：一个服务器的崩溃会造成下一个服务器的崩溃
  • Heisenbug：啥？？？（薛定谔的bug，不同观测表现不同）
  • Customer Reported failure：客户发现的错误
  • Time Travel：大概是全球时区不同
  • Owned：一些节点的安全性被攻破
  • Natural Disease：不可抗自然灾害
  • Police Raid：警方查水表
  • Operator Error：拥有root权限的临时工删数据库…
  • Runaway Automation：拥有root权限的临时工挂机自动了…
  • Certificate Expires：证书到期
  • Fall to pay the bill：没钱了
  • Non-linear response to change：小的扰动造成大的问题
  • Peformance Cliffs：性能滑坡
  • Very Important User：
  • Non-hermetic Builds：非封闭的构建（部分src找不到）
  • Firewall：突然多了一堵墙
  • Kernel Memory Leak：核心内存不足
  • Isolation Failures：
  • Hash Collision：哈希值冲突（数据量很多时
  • Incorrect Algorithm

L5: The many types of fail

解决Failure的方式：Divide and conquer

  • Split failure into categories
  • Design solutions for important categories
  • Importance = frequency * impact
  • Ideally measure frequency and impact
    • Otherwise, guess

What is the useful categories？

1. Network failure vs. Node failure
Network Failures：
  • Protocols提供有用的抽象
  • TCP/IP决定一个nodes pair能否进行通信
  • SSH提供了节点间连接的安全性

So还有哪些情况是需要我们handle的？

  1. Loss of connectivity：可能会丢包、断开连接
  2. Network not fast enough：会让人不知道是网络连接慢还是远程节点本身速度慢
  • Loss of connectivity可能会造成Network Patition：

Network Patition： At least two components of system continue to run, but can’t communicate.

Why do we care about patitions？

  • 如果我们的分布式系统是只读的，节点之间很少需要进行通信，彼此的state不经常发生变化，那么不需要关心patition；
  • 但是大多数的分布式系统的state会发生改变，所以需要保证所有节点的state保持一致——
  • 当Network patition发生时，两个Nodes在不同的subgraphs对同一项数据进行写入（如两个用户对同一条FaceBook post进行评论，那么两个subgraphs共享的state就会diverge——
  • You have lost consistency.
  • 当然，也可以选择在diverge之后重新合并它们的内容，但是这样的做法很烦，更好的做法是maintain a strict consistent system。

“Safe Answer” to network patition

  1. 检测partition
  2. 除了一个subgraph之外，disable所有其他的nodes
  • 这样的做法保持了consistency，at the cost of availability。一旦某个中间Nodes断开连接，那么就会造成比较小的那个subgraph中所有nodes被taken down；
  • 这样的trade-off被称作the CAP principle (consistency, availability and patitioning: you can only have two of three)。

回到我们简单的例子中，我们设计一个简单的算法来检测patition：

Each node:

  1. Ping all N nodes
  2. Count Responses R
  3. If R <= floor(N/2), node halts

当某个节点发现自己收到的响应少于总数的一半时，说明该节点处于某个被分离的subgraph，此时应进入read-only的安全模式，等待连接恢复。

但是该算法在3个及以上的subgraphs的时候无能为力，可能会导致所有的nodes都shut-down

Node Failures：
  • Fail stop：如崩溃、电源不足、硬件故障、内存/磁盘空间不足等
  • Strategies：
    • 保存checkpoint，然后重启——可能导致high latency
    • 将该节点的工作复制到另一节点上，代替进行工作——high cost
2. Byzantine Failure：
  • 所有不属于Fail stop的错误都属于拜占庭错误，如：
    • 内存上或磁盘上的某个bit发生翻转，导致数据错误
    • 过时的代码让node发送了无效的信息
    • node运行了恶意软件等
  • Byzantine式的错误太多，以至于不可能建立统一的解决方法
  • 解决方式：将Byzantine Fail转化成Fail stop
    • Checksum
    • Assertions
    • Timeouts

Summary

L6：Byzantine Fault Tolerance

Byzantine Failure：
  • 所有非“Fail stop”类型的错误；
  • 由背叛者节点Traitor nodes发送与正常节点相矛盾的信息，造成不正确的结果；背叛者节点分为混乱节点Flaky nodes（意外的数据错误）与恶意节点Malicious nodes（恶意软件攻击）。

Why study Byzantine failure?

  • 极端情况：bitcoin（各节点互不信任）、波音777（不容一点差错）
  • 这个问题很有趣，从1980s就吸引了很多研究者的关注

Byzantine Generals：拜占庭将军问题

两个将军约定某个时刻对敌人发动进攻，如果达成了共识他们就能赢，反之则会被地方打败——

  • TCP/IP协议的三次握手方案不能保证双方达成共识！因为最后一次信息传输总有出错的可能

拜占庭将军问题由1982年的一篇论文提出：The Byzantine Generals Problem, Leslie Lamport, Robert Shostack and Mashall Peace. ACM TOPLAS 4.3, 1982.
